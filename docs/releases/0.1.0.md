## Summary

PLC Copilot 0.1.0 is the first public npm release of the PLC Copilot
industrial code-generation toolchain.

This release publishes the core PIR model, the vendor-neutral codegen
core, the Siemens / CODESYS / Rockwell backend packages, and the CLI
package. Everything ships under the `next` dist-tag — promotion to
`latest` is intentionally deferred until after a human inspection
period; see _§Do not promote to latest yet_ below and §5 of the
postmortem.

## Included capabilities

- PIR (Project Intermediate Representation) schema + validation
  primitives (`@plccopilot/pir`).
- Vendor-neutral compiler / codegen core
  (`@plccopilot/codegen-core`).
- Siemens TIA Portal SCL generation
  (`@plccopilot/codegen-siemens`).
- CODESYS Structured Text generation
  (`@plccopilot/codegen-codesys`).
- Rockwell-flavoured ST + UDT generation
  (`@plccopilot/codegen-rockwell`).
- CLI (`@plccopilot/cli`) commands:
  - `generate`
  - `validate`
  - `inspect`
  - `schema`
  - `help`
- Published JSON Schemas (CLI machine output contracts):
  - `cli-result.schema.json`
  - `serialized-compiler-error.schema.json`
  - `generate-summary.schema.json`
  - `web-zip-summary.schema.json`

## Verification before publish (Sprint 67 preflight)

Operator ran locally and signed off before triggering the workflow:

```sh
pnpm run ci
pnpm release:publish-dry-run
pnpm consumer:install-smoke
pnpm release:provenance
pnpm release:publish-real --validate-only --version 0.1.0 --tag next
```

Then via GitHub Actions:

- _Actions → Publish packages → Run workflow_ with `dry_run: true`
  passed.
- The `plccopilot-publish-preflight-tarballs` artifact contained the
  expected six `.tgz` + `manifest.json`.

## Verification after publish (Sprint 67 close-out)

Post-publish gate ran via _Actions → Post-publish verify → Run workflow_
with `version: 0.1.0`, `registry: https://registry.npmjs.org`,
`npm_tag: next`, `check_tag: true`. The same three checks were also
re-run locally on the operator's machine for sign-off:

```sh
pnpm release:provenance    --version 0.1.0
pnpm release:npm-view      --version 0.1.0 --tag next
pnpm release:registry-smoke --version 0.1.0
```

Results — three layered checks all passed:

| Check | Status | What it proved |
| --- | --- | --- |
| `release:provenance` stub | ✅ passed | Publish workflow grants `id-token: write`, `--provenance` is hardcoded in both the workflow YAML and the publish argv builder. |
| `release:npm-view` (with tag) | ✅ passed | All 6 candidates resolve on the registry with the expected name + version + `dist.tarball` + `dist.integrity`; the `next` dist-tag resolves to `0.1.0` for every package. |
| `release:registry-smoke` | ✅ passed | A fresh consumer project under `os.tmpdir()` `npm install`s `@plccopilot/cli@0.1.0` from the registry, runs `help` / `schema --name` / `schema --check` / `totally-unknown --json` / `inspect --json` / `validate --json` / `generate --backend siemens --json`. The CLI produced Siemens artifacts end-to-end from the registry-installed bin. |

This is the first proof that an external consumer can install and use
the `@plccopilot/*` packages from the public registry.

## Known limitations

- First release is published under the `next` dist-tag, **not**
  `latest`. Promotion is recorded as a deferred decision in §5 of the
  [postmortem](../first-publish-postmortem.md); it remains a follow-up
  sprint.
- The provenance stub verifies the publish path is *configured* for
  provenance and the registry minted attestations during the run.
  Deep npm provenance attestation verification (Sigstore Fulcio chain
  walk + OIDC claim matching against this repo + workflow) is **not**
  implemented yet — that is reserved for a future sprint.
- No git tag (`v0.1.0`) and no GitHub Release have been created
  automatically; the postmortem flags those as manual after-action
  items.
- The web app (`@plccopilot/web`) is intentionally not on this list
  — it is an app, not a publishable library.

## Do not promote to latest yet

Promotion to `latest` is **intentionally deferred** until a human
inspection period completes (see §5 of the postmortem). The
post-publish verification has passed, but the conservative first
release stays on `next` so a real bug discovered in the field would
not poison consumers running `npm install @plccopilot/cli`. When the
decision flips, the operator runs:

```sh
npm dist-tag add @plccopilot/pir@0.1.0            latest
npm dist-tag add @plccopilot/codegen-core@0.1.0   latest
npm dist-tag add @plccopilot/codegen-codesys@0.1.0 latest
npm dist-tag add @plccopilot/codegen-rockwell@0.1.0 latest
npm dist-tag add @plccopilot/codegen-siemens@0.1.0 latest
npm dist-tag add @plccopilot/cli@0.1.0            latest
```

A future sprint will move that behind the same `workflow_dispatch`
gate.
