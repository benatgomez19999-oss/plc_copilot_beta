## Summary

PLC Copilot 0.1.0 is the first public npm release of the PLC Copilot
industrial code-generation toolchain.

This release publishes the core PIR model, the vendor-neutral codegen
core, the Siemens / CODESYS / Rockwell backend packages, and the CLI
package. The initial publish (Sprint 67) staged everything under
`next` to allow a human inspection period; the `latest` dist-tag was
promoted in Sprint 68 closeout once the post-publish verification and
inspection both passed. Both `next` and `latest` now resolve to
`0.1.0` on the registry.

```sh
pnpm release:provenance    --version 0.1.0
pnpm release:npm-view      --version 0.1.0 --tag next
pnpm release:registry-smoke --version 0.1.0
```

Results — three layered checks all passed:

| Check | Status | What it proved |
| --- | --- | --- |
| `release:provenance` stub | ✅ passed | Publish workflow grants `id-token: write`, `--provenance` is hardcoded in both the workflow YAML and the publish argv builder. |
| `release:npm-view` (with tag `next`) | ✅ passed | All 6 candidates resolve on the registry with the expected name + version + `dist.tarball` + `dist.integrity`; the `next` dist-tag resolves to `0.1.0` for every package. |
| `release:registry-smoke` | ✅ passed | A fresh consumer project under `os.tmpdir()` `npm install`s `@plccopilot/cli@0.1.0` from the registry, runs `help` / `schema --name` / `schema --check` / `totally-unknown --json` / `inspect --json` / `validate --json` / `generate --backend siemens --json`. ## Known limitations

- **Cryptographic Sigstore-bundle verification is not implemented yet.**
  Sprint 70 verifies the attestation *claims* (subject + workflow +
  repository + recorded git commit) end-to-end, but does **not**
  walk the Fulcio certificate chain, validate the Rekor inclusion
  proof, or verify the bundle signature. The report explicitly
  records `cryptographic_verification.implemented: false`. For an
  npm-supported cryptographic check today, run
  `npm audit signatures` against an installed copy of these
  packages — that is independent of this tool. Full Sigstore
  verification is reserved for a future sprint.
- The web app (`@plccopilot/web`) is intentionally not on this list
  — it is an app, not a publishable library.

## Promotion to `latest` (Sprint 68)

The initial publish staged the release under `next` so a real bug
discovered in the field would not poison consumers running the
default `npm install @plccopilot/cli`. After the human inspection
period passed without consumer-facing reports, Sprint 68 ran
`pnpm release:promote-latest` via the `Promote latest` workflow
(behind the `npm-publish` environment + the `promote @plccopilot 0.1.0
to latest` confirmation string) to move the `latest` dist-tag.

The runner is dist-tag-only by construction — `assertNoPublishSurface`
on every spawn, no `npm publish` shell line in the workflow YAML,
parser rejects `--publish` / `--dry-run` / `--no-dry-run` / `--yes`
at parse time. No tarballs were rebuilt or republished.

Effective registry state after the promotion (verified via
`pnpm release:npm-view --version 0.1.0 --tag latest`):

```
@plccopilot/pir@latest             → 0.1.0
@plccopilot/codegen-core@latest    → 0.1.0
@plccopilot/codegen-codesys@latest → 0.1.0
@plccopilot/codegen-rockwell@latest → 0.1.0
@plccopilot/codegen-siemens@latest → 0.1.0
@plccopilot/cli@latest             → 0.1.0
```

`@next` continues to point at `0.1.0` for every candidate — both tags
agree on the same version.

### Workflow inputs (verbatim, for the postmortem record)

```
version:  0.1.0

registry: https://registry.npmjs.org

```

The `confirm` string matches `expectedGithubReleaseConfirmation('0.1.0')`
byte-for-byte; the runner refused to start without it. The runner
also ran `assertNoNpmMutationSurface` against the `gh` argv before
spawn — confirming no `publish` / `dist-tag` / `npm` token was ever
present in the spawned argv (defence-in-depth, unit-tested).
