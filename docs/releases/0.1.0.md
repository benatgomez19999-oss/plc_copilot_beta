## Summary

PLC Copilot 0.1.0 is the first public npm release of the PLC Copilot
industrial code-generation toolchain.

This release publishes the core PIR model, the vendor-neutral codegen
core, the Siemens / CODESYS / Rockwell backend packages, and the CLI
package. Everything ships under the `next` dist-tag — promotion to
`latest` is intentionally deferred until after the registry install
smoke and a human review pass.

## Included capabilities

- PIR (Project Intermediate Representation) schema + validation
  primitives (`@plccopilot/pir`).
- Vendor-neutral compiler / codegen core
  (`@plccopilot/codegen-core`).
- Siemens TIA Portal SCL generation
  (`@plccopilot/codegen-siemens`).
- CODESYS Structured Text generation
  (`@plccopilot/codegen-codesys`).
- Rockwell-flavoured ST + UDT generation
  (`@plccopilot/codegen-rockwell`).
- CLI (`@plccopilot/cli`) commands:
  - `generate`
  - `validate`
  - `inspect`
  - `schema`
  - `help`
- Published JSON Schemas (CLI machine output contracts):
  - `cli-result.schema.json`
  - `serialized-compiler-error.schema.json`
  - `generate-summary.schema.json`
  - `web-zip-summary.schema.json`

## Verification before publish

Operator runs locally and signs off before triggering the workflow:

```sh
pnpm run ci
pnpm release:publish-dry-run
pnpm consumer:install-smoke
pnpm release:provenance
pnpm release:publish-real --validate-only --version 0.1.0 --tag next
```

Then via GitHub Actions:

- _Actions → Publish packages → Run workflow_ with `dry_run: true`
  (default), `version: 0.1.0`, `npm_tag: next`, `confirm:` empty.
- Inspect the `plccopilot-publish-preflight-tarballs` artifact.

## Verification after publish

After the real publish workflow run completes, verify from a clean
consumer context:

```sh
pnpm release:provenance --version 0.1.0
pnpm release:npm-view    --version 0.1.0 --tag next
pnpm release:registry-smoke --version 0.1.0
```

Or _Actions → Post-publish verify → Run workflow_ with the same
inputs (`version: 0.1.0`, `npm_tag: next`, `check_tag: true`).

## Known limitations

- First release is published under the `next` dist-tag, **not**
  `latest`. Promotion is deferred until post-publish review passes.
- Deep npm provenance attestation verification (Sigstore Fulcio chain
  walk + OIDC claim matching) is not implemented yet; Sprint 65's
  `release:provenance` stub verifies the publish workflow is
  *configured* for provenance, locally.
- No git tag or GitHub Release is created automatically yet.
- The web app (`@plccopilot/web`) is intentionally not on this list
  — it is an app, not a publishable library.

## Do not promote to latest yet

Promotion to `latest` is **intentionally deferred** until after:

1. `pnpm release:npm-view --version 0.1.0 --tag next` passes.
2. `pnpm release:registry-smoke --version 0.1.0` passes from a fresh
   consumer.
3. A human reviewer signs off in
   [`first-publish-postmortem.md`](../first-publish-postmortem.md).

When that condition is met, promotion is one `npm dist-tag add` per
candidate (see the postmortem template). A future sprint will
automate the promotion behind the same `workflow_dispatch` gate.
